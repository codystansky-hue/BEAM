## Commands

**Run the Trame app (active):**
```bash
.\.venv\Scripts\python.exe trame_app.py
```

**Run the Streamlit app (legacy):**
```bash
.\.venv\Scripts\python.exe -m streamlit run app.py
```

**Install Python dependencies:**
```bash
pip install -r requirements.txt
```

**Run all tests:**
```bash
.\.venv\Scripts\python.exe -m pytest tests/
```

**Run a single test file:**
```bash
.\.venv\Scripts\python.exe -m pytest tests/test_mesher.py -v
```

## Architecture

This is an engineering desktop app for composite beam cross-section analysis. It has two front-ends:

- **Streamlit** (`app.py`): Original UI. Run with `streamlit run app.py`.
- **Trame/Vue3** (`trame_app.py` + `trame_app/`): New UI with integrated 3D VTK visualization. Run with `python trame_app.py`. This is the active development target.

Both front-ends drive the same pipeline of external solvers and a pure-Python mesher. Material data is persisted in local JSON files in `db/`.

### Trame app structure (`trame_app/`)

- `engine.py`: Async execution pipeline orchestrating solver runs
- `state.py`: State management, DB load/save, defaults
- `vtk_views.py`: PyVista/VTK visualization helpers
- `pages/preprocessing.py`: Material/geometry input UI
- `pages/solution_setup.py`: Solver configuration
- `pages/results.py`: Results display
- `pages/visualization.py`: 3D view controls

### Analysis Pipeline (in execution order)

1. **Meshing** (`mesher.py` → `ProfileMesher`): Reads a 2D spine curve from a `.3dm` Rhino file using `rhino3dm`. Generates a structured transfinite quad mesh by offsetting the curve inward by the total layup thickness. Outputs `mesh_data` dict with keys `nodes` (Nx3 numpy array, in mm), `elements` (Mx4, 0-based), and `tangents` (Mx3). Uploaded `.3dm` files are saved to `meshes/last_uploaded.3dm`.

2. **Material property assignment** (`materials.py` → `assign_properties_to_mesh`): Takes `mesh_data` and a `Layup` object, and computes the global material orientation angle per element (local spine tangent + user ply angle). Returns `element_properties` list consumed by solvers.

3. **Cross-section analysis** — one of:
   - **SwiftComp** (`solvers.py` → `SwiftCompSolver`): Writes a `.sc1` deck, runs `SwiftComp.exe`, and parses the 6×6 stiffness matrix `K`.
   - **VABS** (`solvers.py` → `VABSSolver`): Writes a `.dat` deck, runs `vabs.exe`, and parses the `.K` file using the `sgio` library.

4. **Global beam analysis** (`solvers.py` → `GXBeamSolver`): Writes a JSON input file, calls `gxbeam_solver.jl` via Julia subprocess (using `Project.toml`/`Manifest.toml` in the project root). Julia script writes a JSON output and a VTK polyline. Julia's `cwd` must be the project root where `Project.toml` lives, not the runs output directory.

5. **Euler buckling** (calculated inline in `app.py`): Uses EI values from the 6×6 stiffness matrix with standard Euler formula.

6. **Local buckling** (`solvers.py` → `CalculiXSolver`): Extrudes the 2D mesh into a 3D C3D8 brick model, writes a CalculiX `.inp` deck with a `*BUCKLE` step, runs `ccx_MT.exe`, and parses the buckling factor from the `.dat` file. Also writes VTK files for ParaView via `write_mode_vtk` (reads eigenmode displacements from the `.frd` file).

### Key data flow

- Material properties are stored in SI units internally (Pa, kg/m³) and converted to GPa/MPa only for UI display.
- Cross-section geometry (mesh nodes) is in **millimeters** throughout the Python layer. CalculiX also receives mm. GXBeam/Euler use meters (converted before passing to Julia).
- CalculiX material input requires MPa (divide Pa by 1e6), and density in tonnes/mm³ (divide kg/m³ by 1e9).
- Run outputs (solver input/output files, VTK exports) are written to timestamped subdirectories under `runs/`.

### Database (`db/`)

JSON flat-file database:
- `fibers.json`: Raw fiber constituent properties (E11_t, E11_c, E22, G12, nu12, density, Xt, Xc, cte11, cte22)
- `resins.json`: Resin/matrix properties (E_t, E_c, nu, density, cte, Xt, Xc, S)
- `laminae.json`: Homogenized lamina properties generated by the micromechanics tool
- `layups.json`: Saved stacking sequences (list of {name, angle} plies)

### Micromechanics (`micromechanics.py`)

`calculate_lamina_properties(fiber, resin, vf)` uses rule-of-mixtures (longitudinal), Chamis semi-empirical (transverse/shear), and Schapery's equations (CTE) to compute orthotropic lamina properties from fiber + resin constituents at a given fiber volume fraction.

### External solvers

Default executable paths (configurable in the UI sidebar):
- SwiftComp: `Swiftcomp/SwiftComp.exe`
- Julia (GXBeam): `julia` (must be on PATH)
- CalculiX: `CalculiX-Windows/bin/CalculiX-2.23.0-win-x64/bin/ccx_MT.exe`

### Test structure

Tests live in `tests/`. They mock `rhino3dm` (an external C extension) using `unittest.mock.MagicMock` and add the parent directory to `sys.path` manually since there is no package `__init__.py`.

### File notes

- `app.py` is the Streamlit entry point; `trame_app.py` is the Trame entry point (active development).
- `gxbeam_solver.jl` calls `Pkg.activate(".")` so it must be invoked with `cwd` set to the project root.
- `pdf_parser.py` uses regex on extracted PDF text to pull material properties from manufacturer datasheets.
- `patch_db.py` and `repair_database.py` are one-off database maintenance scripts.
- `run_mesher.py` is a standalone mesher test/debug script.
